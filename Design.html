Assignment 2: Design

Design of the Databases

The databases for the assignment are completely implemented in PostgreSQL. There are three global databases, namely dist_queue (Broker),  dist_queue_2 (Write Broker Manager) and dist_queue_3 (Read Broker Manager). These databases were designed keeping in mind the different design choices for the components of the Distributed Queue.

Design of the Broker Database (dist_queue)

The dist_queue database has the following tables:

  • One table to store the metadata of all the partitions (all_partitions)
  • One table per partition inside the broker

Each broker has a separate instance of dist_queue.

Metadata of all partitions (all_partitions)

The all_partitions table is responsible for storing the metadata for all the partitions kept in a broker. In the design, each partition is assigned to exactly one topic. One topic may have many partitions, kept in different brokers. This has been done keeping in mind easier implementation. Multiple topics inside a partition will warrant additional bookkeeping in the message table (we would have to store the topic name as well), increasing the amount of storage required because the message queue can reach millions of messages.

Design of the Write Broker Manager Database (dist_queue_2)

The dist_queue_2 database stores five tables for metadata pertaining to the Write Broker Manager:

  • One table for information about all topics (all_topics)
  • One table for information about all consumers (all_consumers)
  • One table for information about all producers (all_producers)
  • One table for information about all brokers (all_brokers)
  • One table for information about all managers (all_managers)

There is only one write broker manager and hence there is only one instance of this database.

Metadata of all topics (all_topics)

Storing information about a topic is necessary for the write broker manager. This is because the presence of a topic should be checked before doing anything.

Since the verification of the consumer and the producer IDs is done at the manager's side only, additional information about the number of consumers and producers is maintained. To direct producer queries, information about the partitions must be maintained.

Keeping these details in mind, the design is as follows:

Metadata of all consumers (all_consumers)

The all_consumers table is used to store information about every consumer.

For each consumer, we need to store an offset into each partition (which belongs to a given topic), till which the consumer has read messages because brokers are like black boxes which do not know anything about consumers.

This offset must be referenced by the id of the partition, thus it is most ideally kept in a JSON format.

What is the purpose of the lastindex column?

The lastindex column is very important to understand how messages are consumed. The consumer can choose to withdraw messages from a given partition or withdraw them from any partition. lastindex shows the partition from which the last message was withdrawn. Suppose the consumer wants messages to come from any partition. To prevent messages coming from the same partition, we cycle over the partitions and deliver messages.

Partition cycling (round robin) starts from (lastindex + 1) % num_partitions. Whenever a partition with a non-delivered message is found, a message is consumed and returned. If no such partition exists, there is no message for this consumer.

Check Design of the Partitions for more information.

Metadata of all brokers (all_brokers)

Information about all the brokers needs to be maintained:

  • While creating a new topic (and distributing partitions for the same across multiple brokers)
  • While sending a message (the load on the broker is increased, and this is maintained in the form of the total number of messages in the broker, read more in Assignment of the Partitions to the Brokers)
  • For recording the broker heartbeats
  • Trivially, for creating new brokers

The all_brokers table serves this very purpose.

Design of the Read Broker Manager Database (dist_queue_3)

The read broker manager database is the same as the write broker manager database (dist_queue_2). This is done keeping in mind the promotion of a read manager to a write manager in the future incase of the failure of a write manager.

Currently, the write manager is an irreplaceable component of the distributed queue and its failure is the failure of the entire distributed queue.

However, the functionalities of the read and write managers have considerable differences.

Generation of Consumer and Producer IDs

The system requires validating the consumer and the producer IDs for each query to see if they subscribe to a particular topic or not. The producer and consumer IDs are hence generated in the following manner:

 

The last digit is 0 or 1 depending on whether it is a consumer or a producer respectively. We then reserve a configurable number of digits to include the topic ID. We run a serial number for each topic and sequentially assign it to each new producer/consumer. We can now check whether the ID corresponds to the particular topic by verifying the digits corresponding to the topic ID.

To see if the client had already registered we simply verify if the digits corresponding to the serial number are less than the current running serial number.

Design of the API Endpoints

Design of the Broker APIs (/broker/main.py)

Creating a new partition (/partitions)

Method: POST

Input format:

        topic: str,

partition: int

Output format:

        status: str (“success” OR “failure”),

        message: str

What does it do?

This API call creates a new partition for a given topic inside the broker. It follows the following steps in order:

  1. UPDATE the all_partititions table, increase the partitionid by 1 for this topic.
  2. Creates the message table for the partition with format <topicname>_<partitionid>

Adding a message (/enqueue)

Method: POST

Input format:

        data: str,

topic: str,

partition: int,

message: str

Output format:

        status: str (“success” OR “failure”),

        message: str (optional, only in the case of “failure”)

What does it do?

This API call adds a new message in the given partition in the broker. It follows the following steps in order:

  1. Get the tailid of the specific partition.
  2. Insert message with messageid = tailid into the partition.
  3. UPDATE all_partitions SET tailid = tailid + 1

Removing a message (/dequeue)

Method: POST

Input format:

        data: str,

partition: int,

offset: int

Output format:

        status: str (“success” OR “failure”),

        message: str

What does it do?

This API call gets a message from the given offset from the partition queue. If there is no such message, then the queue is empty and the same is returned as a message. Otherwise, the message itself is returned.

Design of the Write Broker Manager APIs (/broker-manager/main.py)

Creating a new topic (/topics)

Method: POST

Input format:

        data: str,

        partitions: int (optional)

Output format:

        status: str (“success” OR “failure”),

        message: str

What does it do?

This API call creates a new topic:

  1. Check if topic is present in all_topics
  2. If YES: return ServerError
  3. If NO:
  1. Is partitions specified?
  1. YES:

                         part = partitions

  1. NO:

                        part = 1

  1. For each partition in part:
  1. Select a lowest load broker
  2. Create a partition on that broker
  1. Update all_topics with new data

Getting the list of all topics (/topics)

Method: GET

Output format:

        status: str (“success” OR “failure”),

        topics: list

What does it do?

This call returns the list of all topics. This is used by the clients (consumers and producers) to get the list of all the topics which they can produce from and then consume from.

Method: POST

Input format:

        topic: str

Output format:

        status: str (“success” OR “failure”),

        consumer_id: str

What does it do?

This call creates a new consumer and returns the consumer ID. To see how the consumer ID is generated, go to generation of consumer and producer IDs. If this topic is not present in all_topics, a ServerError is returned.

Method: POST

Input format:

        topic: str

Output format:

        status: str (“success” OR “failure”),

        producer_id: str

What does it do?

This function is similar to registering a new consumer, except that if the topic is not present already, it will be created by calling the creating a new topic routine.

Producing a message (/producer/produce)

Method: POST

Input format:

        topic: str

        message: str

        partition: int (optional)

        producer_id: str

Output format:

        status: str (“success” OR “failure”)

What does it do?

This function allows a producer to send a message to a specific topic. The function follows the steps:

  1. Check if this producer is actually subscribed to the topic, otherwise return a ServerError.
  2. If a partition is specified, then check if this partition is existent, if it is then send a message to this partition, else if the partition is non existent return a ServerError
  3. If no partition is specified, then assign messages randomly to any active partition.

Synchronization (/sync)

Method: POST

Input format:

        query: string,

        sender: string

Output format:

        status: str (“success” OR “failure”)

What does it do?

This function call is very important. The sql query on the sender end is converted into a query string. This string is then broadcasted to every broker manager (except the write manager). The read managers on receipt of this query, simply execute and commit into their database and send the “ACK” message as response.

Method: GET

What does it do?

This call is for consumers only. It reveals the number of pending messages in all the partitions of this topic.

Design of the Partitions

The number of partitions for a topic is to be specified while creating the topic (defaulting to 1 if not specified). The partitions are assigned to the active brokers in a round robin manner. If a broker holding a partition is inactive, and a client tries to access it, we return an error message specifying that the partition is inactive. We do not migrate the partitions because it is assumed that the only time we will need to do it is when the broker hosting the partition is inactive in which case we have no means of accessing the partitions. Both producers and consumers can specify which partition they want to write into and read from respectively. However if nothing is specified, the message is produced randomly into any of the active partitions and is read from the partitions in a round robin manner. To read in a round robin manner, we store a lastindex variable for each consumer to remember the partition it last read from.  

Assignment of the Partitions to the Brokers

Assignment of the partitions to the brokers follows selecting the broker with the least load first.

Design of Write Ahead Logging (WAL)

We are using PostgresDB to emulate persistent storage and observed that Postgres inherently implements the WAL mechanism which can be configured to be turned on or off and is on by default. Hence, we have not implemented WAL additionally in our application as Postgres takes care of it internally on its own.

Creating and Removing Brokers

We believe that the administrator must have control over the creation and the removal of brokers and this should not be the responsibility of the users (producers and consumers) interacting with the broker manager. For this, some additional endpoints have been created.

Need of Synchronization

Since the clients can contact any broker manager, it is important that metadata at all the managers be the same. For example, if a consumer contacts a broker manager to retrieve from a partition, the offset will be uploaded on that particular manager. Now next time if it contacts another broker manager, the next message should be retrieved, meaning this broker manager should also have the updated offset.

Synchronizing Mechanism

Assumptions: No faults, i.e., the broker managers will always be up and will always be able to communicate reliably with each other

Mechanism:

  • The broker manager where the change originates will broadcast the write query to all the other broker managers
  • The broker managers receiving the query will make the change and commit it in their respective database and send back an ACK
  • The original broker manager will then commit the query in their database

How are new Read Managers Added?

Our implementation does not provide the possibility of adding new write managers. There is only a single write manager. However, when new read managers are added, their database should be exactly the same as the databases of all the other managers.

In this case, it is assumed that the write manager is supreme and has the correct form of the database.

The write manager sends its tables in entirety to the newly created read manager which has an empty database. There is a separate API call for this inside the read manager file (read.py).
